By default, the Portal Controller generates entities straight into the writer, no customization needed. If you want to shape the entities a Promise (and its Resources) publish to
a portal, you can bind it to a `PortalCustomization`.

## Creating a `PortalCustomization`

`PortalCustomization` (`portal.kratix.io/v1alpha1`) is a cluster-scoped resource:

```yaml
apiVersion: portal.kratix.io/v1alpha1
kind: PortalCustomization
metadata:
  name: backstage-add-team-owner
spec:
  portal: backstage # required: the portal type this profile applies to
  promise:
    containers:
      - name: add-owner
        image: acme/backstage-owner-injector:v1
        args: ["--field=spec.owner", "--value=platform-team"]
  resource:
    containers:
      - name: add-tags
        image: acme/backstage-tag-injector:v1
```

- **`spec.portal`** (required) is the portal type this profile applies to, for example `backstage`. It is never a specific portal, credential, or URL, those live on the
  `SKEIntegration`.
- **`spec.promise.containers[]`** runs, in order, for the Promise-level sync.
- **`spec.resource.containers[]`** runs, in order, once per bound Resource request.

Both lanes run their containers between `generate` and `writer`:

```
generate  →  customize (this profile's containers, in order)  →  writer
```

Containers use the same shape as any Kratix pipeline container (`image`, `args`, `command`, `env`, `envFrom`, `volumeMounts`, `resources`, `securityContext`). They edit the entity
documents in place and may read the parent object, the same way `generate` does. They do not receive the portal credential, that is only ever injected into the writer container.

:::tip

`PortalCustomization` resources are cluster-wide. Give them unique, descriptive names.

:::

## Binding a Promise to a `PortalCustomization`

Label the Promise with the profile's name:

```yaml
metadata:
  labels:
    kratix.io/portal-customization: backstage-add-team-owner
```

The label alone is enough to opt in, the portal type is derived from the profile's own `spec.portal`, so you do not also need a `kratix.io/portal-<type>` label unless the Promise
also needs to opt into a specific portal instance.

If you have more than one `PortalCustomization` you want to apply in different contexts, use an instance suffix the same way as portal binding labels:

```yaml
metadata:
  labels:
    kratix.io/portal-customization.staging: backstage-staging-customization
```

## Worked example: setting owner and tags

**Patching the generated Backstage `catalog-info.yaml` in place, for every Resource of a bound Promise:**

```yaml
apiVersion: portal.kratix.io/v1alpha1
kind: PortalCustomization
metadata:
  name: cicd-backstage-customization
spec:
  portal: backstage
  resource:
    containers:
      - name: set-owner-and-tags
        image: ghcr.io/syntasso/kratix-pipeline-utility:v0.0.1
        command: ["sh", "-c"]
        args:
          - |
            set -e
            file=/kratix/output/catalog-info.yaml
            yq -i '.spec.owner = "platform-team"' "$file"
            yq -i '.metadata.tags = (.metadata.tags // []) + ["tier-1", "managed-by-kratix"]' "$file"
```

Bind the Promise to it:

```yaml
metadata:
  labels:
    kratix.io/portal-customization: cicd-backstage-customization
```

## Status

A `PortalCustomization` reports a single `Accepted` condition, this reflects whether the resource itself is structurally valid, not whether any particular Promise's binding to it
resolved. Whether a Promise's binding resolved is reported on the Promise, via the `PortalBindingReady` condition described in [Installation](./installation#bind-a-promise-to-a-portal).
