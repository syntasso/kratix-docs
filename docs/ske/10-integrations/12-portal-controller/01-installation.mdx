## Install the Portal Controller

:::important

The `portal-controller` integration type requires the SKE Operator version that introduced it, check the [SKE Operator release notes](../../releases/ske-operator) for the
minimum version.

:::

The Portal Controller is installed and upgraded by the [SKE Operator](../../installing-ske/advanced-install) from a single [`SKEIntegration`](../../reference/ske-operator#skeintegration-crd)
resource of `spec.type: portal-controller`. Only one `SKEIntegration` of this type may exist on a platform at a time.

Applying the resource installs the Portal Controller Deployment, the `PortalCustomization` Custom Resource Definition, and the RBAC and Service Account it runs as, into the
`kratix-platform-system` namespace.

## Connect a portal

Each entry under `spec.portals` describes one portal instance:

| Field | Required | Meaning |
|---|---|---|
| `name` | yes | Identifies the portal. Must be unique within `type` (you can have two Backstage portals, but not two both named `prod`). |
| `type` | yes | The portal type this entry targets, for example `backstage` or `cortex`. |
| `url` | depends on `type` | The portal's base URL. |
| `secretRef` | depends on `type` | Names the portal's credential Secret (see [Credential Secret handling](#credential-secret-handling)). |
| `config` | depends on `type` | Portal-specific settings. Values may be strings or objects/arrays. |
| `adapterConfig` | no | Overrides the bundled default adapter image for this one portal, and/or passes portal-wide `generateFlags` to it. See [Adapter image resolution](#adapter-image-resolution) and [Portal-wide generate flags](#portal-wide-generate-flags). |

You can declare more than one portal, of the same or different types, on the same `SKEIntegration`. What each of these fields needs varies by `type`, Backstage and
Cortex are covered separately below.

### Configuring a Backstage portal

```yaml
apiVersion: platform.syntasso.io/v1alpha1
kind: SKEIntegration
metadata:
  name: portal-controller
spec:
  type: portal-controller
  version: v0.1.0
  portals:
    - name: prod
      type: backstage
```

A Backstage portal needs no `url`, `secretRef`, or `config`, the writer ships entity documents to a Kratix state store rather than calling a live Backstage API, so there is
no credential to configure. You will need a Destination labelled `environment: backstage` for those documents to land on, see
[Creating the Entity State Store](../backstage/installation/creating-entity-state-store#configure-a-destination-for-backstage).

Backstage does not run a setup Job or a status exporter, see [What gets deployed per portal](#what-gets-deployed-per-portal) below for why that only applies to Cortex
today.

### Configuring a Cortex portal

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: cortex-token
  namespace: kratix-platform-system
type: Opaque
stringData:
  token: <cortex-api-token>
---
apiVersion: platform.syntasso.io/v1alpha1
kind: SKEIntegration
metadata:
  name: portal-controller
spec:
  type: portal-controller
  version: v0.1.0
  portals:
    - name: prod
      type: cortex
      url: https://api.getcortexapp.com
      secretRef:
        name: cortex-token
      config:
        repositoryName: syntasso/platform-catalog
        provider: github
        integrationAlias: github-main
        branch: main
```

A Cortex portal requires `url` and `secretRef.name`, and `config` must include:

| Key | Required | Meaning |
|---|---|---|
| `repositoryName` | yes | Git repository the Cortex workflows commit into. |
| `provider` | yes | `github` or `gitlab`. |
| `integrationAlias` | yes | Cortex git integration alias. |
| `branch` | no | Optional branch override. |
| `path` | no | Optional path within the repository. |

Set up your Cortex workspace and git integration as described in [Configuring Cortex](../cortex/configuring-cortex#set-up-the-cortex-workspace) before connecting it here.

## What gets deployed per portal

Every portal syncs Promises and Resources through the same `generate → [customize] → writer` pipeline Jobs described in [Customization](./customization). Cortex
portals additionally run:

- A one-time, portal-scoped **setup Job**, that runs when the portal is created and re-runs when its configuration changes, to idempotently ensure the shared Cortex
  relationship type and entity type this integration depends on exist.
- A long-lived **status exporter**, a Deployment and Service (one replica, `ske-portal-exporter` Service Account) that pushes resource status changes to the Cortex API as they
  happen, rather than waiting for the next sync Job. It is created and owned by the `SKEIntegration`, and removed if the portal is removed.

Backstage does not need either of these, its writer ships straight to a state store with no live API to call, so there is nothing to keep a persistent connection to.

You can tune the Cortex status exporter's pod (and the setup/sync Jobs for that portal) via `portals[].config`:

```yaml
portals:
  - name: prod
    type: cortex
    # ... url, secretRef, repositoryName, provider, integrationAlias as above
    config:
      imagePullSecrets: # extra pull-secret names, always merged with syntasso-registry
        - customer-registry
      deploymentSpec: # partial Deployment spec, merged into the exporter Deployment
        template:
          spec:
            nodeSelector:
              tier: portal
            tolerations:
              - key: dedicated
                operator: Equal
                value: portal
                effect: NoSchedule
```

`deploymentSpec` is strategically merged into the exporter Deployment, prefer pod-level fields such as `nodeSelector`, `tolerations`, and `affinity`. Controller-owned fields (the
exporter container itself, its Service Account, labels, credential injection, and image pull secrets) are always re-applied afterwards, so a `containers` entry in the overlay is
discarded rather than replacing the exporter's own container.

## Credential Secret handling

The Portal Controller does not create or manage a portal's credential Secret, and not every portal needs one, Backstage today does not. Where a portal's `secretRef` is
set (required for Cortex), create the Secret yourself in `kratix-platform-system` before installing or upgrading, with the credential under a `token` key:

```yaml
apiVersion: v1
kind: Secret
metadata:
  name: cortex-token
  namespace: kratix-platform-system
type: Opaque
stringData:
  token: <cortex-api-token>
```

The operator only validates that a referenced Secret exists before installing or upgrading; if it is missing, the `SKEIntegration` reports `IntegrationDeploymentReady: False`
until the Secret is created. Once installed, the credential is injected directly into the sync pipeline's writer container (and, for Cortex, the status exporter) and is never
written to a shared volume or exposed to any other pipeline stage.

## Bind a Promise to a portal

A Promise (or, since the resource lane shipped, a Resource request) opts into a portal with a label:

```yaml
metadata:
  labels:
    kratix.io/portal-backstage: "true" # opt in to the default portal of type "backstage"
```

If you have more than one portal of the same type, name the one you want with an instance suffix:

```yaml
metadata:
  labels:
    kratix.io/portal-backstage.prod: "true" # opt in to the portal named "prod"
```

If a Promise's label matches more than one portal of a type and no instance is given, or matches no portal at all, the Promise reports `Ready: False` on its
`PortalBindingReady` condition with a reason explaining why, rather than silently guessing.

## Adapter image resolution

By default, every portal uses the adapter bundled inside the Portal Controller's own image, at the version set on `spec.version`. To pin a different adapter image for
one portal, for example to bring your own portal integration, set `adapterConfig.image`:

```yaml
portals:
  - name: prod
    type: backstage
    adapterConfig:
      image: acme/custom-portal-adapter
      version: v3
```

`adapterConfig.image` is optional. An entry may set [`generateFlags`](#portal-wide-generate-flags) alone to keep running the bundled default adapter, set `image` (with
`version` appended) to override it, or set both. `adapterConfig.version` may only be set alongside `adapterConfig.image`, there is no version-only override of the bundled default
adapter, and setting `version` without `image` is rejected.

## Portal-wide generate flags

`adapterConfig.generateFlags` is a list of flags passed verbatim to the adapter's `generate` stage for that portal. Every Promise and Resource routed through the portal
gets the same behaviour, so this is a portal-wide setting, not per-Promise. Two flags are currently wired:

| Flag | Portal type | Effect |
|---|---|---|
| `--fixed-namespace=<value>` | `cortex` | Hard-codes `metadata.namespace` to `<value>` and hides the namespace input field on generated workflows. `<value>` must be a valid DNS-1123 label, an invalid value fails the generate stage. |
| `--single-spec-page` | `backstage` | Merges the generated Template's `spec` form pages into one, instead of one page per top-level `spec` property. |

```yaml
portals:
  - name: prod
    type: cortex
    # ... url, secretRef, config as above
    adapterConfig:
      generateFlags:
        - "--fixed-namespace=team-a"
```

These reproduce behaviour that the standalone controllers offered per-Promise (the Cortex `kratix.io/cortex-fixed-namespace` annotation and the Backstage
`singleSpecPage` option), now set once at the portal level. See the [Backstage](./migrating-from-backstage) and [Cortex](./migrating-from-cortex) migration guides for how
the per-Promise forms map across.

:::warning

Do not put credentials in `generateFlags`. Unlike a `secretRef` credential (injected into the writer container via `secretKeyRef`), these flags are passed as container arguments
and are visible to anyone with read access to the Job in `kratix-platform-system`.

:::

## Air-gapped environments

The Portal Controller image can be mirrored to your own registry the same way as other SKE components. See
[Deploying Within Air-gapped Environments](../backstage/air-gapped-installation/intro) for the general pattern of mirroring images and release manifests.
